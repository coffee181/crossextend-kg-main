| T1 | On Aurex BMSCore-410, begin the post-repair check by naming the repaired item, such as controller replacement, sense-wire repair, seal replacement, or CAN-branch correction, together with the original complaint and the exact operating state that exposed it. The final proof has to stay tied to the real failure window. |
| T2 | Inspect the repaired area in static condition first, confirming connector seating, secondary locks, ground-eyelet witness, enclosure seal compression, harness clip placement, and the relationship between the repaired branch and nearby sense headers, NTC leads, or CAN splice points before power is restored. Save the baseline physical state so the later proof can be compared against a real before-and-after condition at the actual battery-service boundary. |
| T3 | Wake the controller and verify supply voltage, CAN presence, representative cell voltages, at least one temperature channel, and the repaired branch behavior before the pack enters a higher operating mode. Watch the live data and the repaired hardware during the same event so the electrical and mechanical evidence stay connected. |
| T4 | Repeat the exact functional comparison relevant to the repaired branch, such as channel-to-channel voltage match at the affected module tap, NTC offset check against a neighboring module face, CAN stability at the BMS branch, or wake timing through the controller supply path, and compare the result with a neighboring healthy reference where appropriate. This turns the closeout into a battery-component proof instead of a generic power-up check. |
| T5 | Hold the pack in the target operating state long enough to see whether the original complaint returns under temperature, vibration, communication activity, or relay transitions, then reinspect the repaired area immediately afterward for connector movement, seal shift, or harness preload. A passing value without a post-event reread is not enough for release. |
| T6 | Finish the post-repair validation only when the record names the repaired component, the operating state used for proof, the observed recovery in live data or command response, and the final physical condition of the adjacent connectors, seals, supports, and battery-specific interfaces around it. The release boundary is a repaired BMS branch, sense path, or enclosure interface that remains coherent electrically and mechanically through the full reproduced operating condition. |
