## Core Constraints
1. The backbone is frozen. Never invent a new backbone concept.
2. You may only choose from these routes: __ALLOWED_ROUTES__.
3. `allow_free_form_growth=__ALLOW_FREE_FORM_GROWTH__`.
4. Return `reuse_backbone` only when the candidate label exactly matches a backbone concept.
5. Otherwise, if the candidate should enter the graph, use `vertical_specialize` with a valid backbone `parent_anchor`.
6. The output must be valid JSON and strictly match the return schema.
7. Reject person names and document titles. They belong in provenance, not as graph nodes.
8. `routing_features.semantic_type_hint` is a soft prior, not a hard label. Use it to improve judgment, but override it when the candidate semantics clearly disagree.
9. `routing_features.is_task_candidate=true` means the candidate is a true step-task node and should normally attach under `Task`.
10. `routing_features.task_step_id` only means the candidate is referenced in a step context if `is_task_candidate` is false; it does not make the candidate a `Task`.
11. If you reject a candidate, you must set `route=reject`, `accept=false`, `admit_as_node=false`, and use exactly one reject reason from this closed set:
    `person_name`, `document_title`, `observation_like_not_grounded`, `cannot_anchor_backbone`, `weak_relation_support`, `low_graph_value`, `unsupported_semantic_type`, `route_not_allowed`, `invalid_backbone_parent`, `llm_no_decision`, `backbone_label_mismatch`.
12. Every candidate includes `prompt_priors`. Treat `prompt_priors.recommended_parent_anchor` as the default starting point for judgment, not as optional noise.

## Attachment Rules
13a. A label that is itself a true O&M step candidate should normally attach under `Task`.
13b. A component, part, connector, hose, clip, manifold, seal, sensor, cavity, guide rail, stud, bead, window, shoulder, edge, fitting, hook, lug, pin, or hardware item should normally attach under `Component`.
13c. A named machine, vehicle, pack, cabinet, line, or platform should normally attach under `Asset`.
13d. An observed result, warning, residue path, leak path, wet boundary, wetting, level, offset, reading, color, angle, depth, height, twist, load, witness, pulsation, or measurement outcome should normally attach under `Signal`.
13e. A persistent condition such as dry, wet after shutdown, fully latched, under circulation, naturally supported, flush, or seated should normally attach under `State`.
13f. A defect, crack, leak, distortion, incomplete insertion, side-load root cause, corrosion source, or failure mode should normally attach under `Fault`.
13g. If `relation_families` contains only `propagation` and/or `communication`, do not attach the candidate under `Task` unless the candidate is a true step-task node.
13h. Being mentioned in a step or having `task_step_id` does not by itself justify `Task`.

## Decision Policy
14. If `prompt_priors.recommended_parent_anchor` is present, default to `vertical_specialize` under that anchor unless an explicit reject rule clearly applies.
15. If `prompt_priors.prior_agreement=true`, rejection should be rare. You need a concrete reason, not just a vague feeling that the label is narrow.
16. Do not reject a candidate solely because it is specific, localized, or step-scoped. Concrete hardware features and repeatable O&M observations are often valid graph nodes.
17. Sparse relation participation alone is not enough reason to reject when semantic type hint, retrieval, and label semantics all support the same backbone anchor. Downstream rule filtering performs the final quality control.
18. `low_graph_value` should be used only for clearly low-value items such as:
    - long clause-like release conditions
    - bookkeeping residue
    - generic placeholders like `identified leak point`
    - explicit history-only context with no stable node semantics
    - purely comparative/meta labels with no reusable industrial entity
19. `observation_like_not_grounded` should be used only when the label is an ephemeral observation clause that cannot function as a stable `Signal` or `State` node even after considering `prompt_priors`.
20. The following usually should be admitted when semantically grounded:
    - `residue color`, `leak path`, `wet boundary`, `coolant level`, `insertion depth`, `latch height`, `hose angle`, `twist`, `side load`
    - `burrs`, `arc marks`, `heat tint`, `pulsation`, `stress whitening`
    - `plate edge`, `tube bead`, `latch window`, `O-ring seat`, `guide rail`, `stud shoulder`, `manifold fitting`
21. The following usually should be rejected:
    - post-repair release-outcome clauses such as `no renewed staining from neck to shield edge`
    - history-only context such as `recent thermal-system repair history`
    - vague meta/comparison labels such as `known-good reference` when they do not denote a stable asset or component

## Quality Rules
22. Observation-like labels may be admitted when they are concrete, repeatable O&M observables, not only when they already look like formal telemetry.
23. Reject low-value fragments, bookkeeping residue, or overly tiny modifiers as `low_graph_value`.
24. Components and other industrial entities should normally be admitted when they clearly anchor to the backbone, even if their relation support is sparse.
25. Historical context may help preserve temporal consistency, but never accept a candidate solely because it appeared before.
